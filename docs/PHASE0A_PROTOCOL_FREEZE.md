# Phase 0A 预注册冻结版

> 冻结说明
>
> 本文为 Phase 0A 方法论预注册冻结版。本文定义的是立项边界、判定逻辑与进入 Phase 0B 的前提，不构成任何 crisis architecture（包括但不限于 override、hard switch、MoE）的立项批准。进入 Phase 0B 之前，必须先完成：benchmark family 锁定、trigger lead-lag audit、bootstrap sign-stability 阈值预注册、以及 2008 OOS 隔离方案固化。

## 1. 核心裁决

当前 Phase 0A 的功能不是设计新模型，而是阻断方法论漂移。当前“现架构 exhausted”的合法边界仅限于：在固定 `Stage A = T5`、冻结 monotone `Stage B`、且只允许低自由度上游凹变换的条件下，现有修补式 `σ_t` 路线已无架构级收益。该结论不能直接外推出“必须进行危机架构重构”。

## 2. 原方案中仍未关闭的方法论漏洞

### 裁决
若无额外协议约束，后续所有架构实验都不具证据资格。

### 依据
1. 缺失 benchmark：在没有 EGARCH/GJR-GARCH 参照时，无法区分 2020 的失败是现架构特有缺陷，还是连续危机波动模型的共同上限。
2. 单一危机样本：只有 2020 而无 2008–2009 / 2000–2002 补窗，不能把 2020 推广为“危机态一般规律”。
3. Trigger 未审计：任何外部信号在未经 lead-lag audit 之前，都不具备进入门控或专家网络的资格。
4. OOS protocol 未锁定：若未预注册，所有 crisis architecture 结果都可被事后重写。
5. `σ_t` 主用途未定义：若主用途漂移，所有 gate 与 success metric 都失去法理基础。

## 3. σ_t 主用途定义

### 裁决
在当前研究生命周期内，`σ_t` 的主用途被唯一且强制地锁定为 A：为 tail family / joint MLE 提供标准化残差输入。实时风控或仓位管理（B）仅被允许作为副用途观察值，绝对不能主导协议或干预评判。

### 依据
选择 A 作为主用途的合法性依据，不是“当前系统已经以下游 density layer 运行为事实”，而是**当前项目的研究目标是恢复下游 density layer / tail / joint MLE 的可行性**。既然当前项目目标是恢复分布层的运转，本协议在逻辑上就必须要求 `σ_t` 以残差白化为主任务：即原始残差 `ε_t` 被白化为同方差且平稳的标准化残差 `z_t`。若允许 B 介入主导，则模型会为迎合短期点估计精度而牺牲残差平稳性，直接破坏恢复联合估计的可能。

若未来项目最高决议将主目标改为实时风险管理（B），则本协议整体作废，评价指标、损失函数与验证协议都必须重写。

## 4. 主用途对应的评价指标优先级与预注册阈值表

### 裁决
所有评估必须按预注册的绝对数值阈值表进行穿透式判定，严格执行阶层式否决。

### 依据（适用对象声明）
本节框架在 **Phase 0A** 的适用对象被明确限定为：**候选模型 = `Stage A = T5`，benchmark = 预先锁定的 EGARCH/GJR-GARCH family**。本节用于回答的是：T5 的失败究竟是其自身特有缺陷，还是连续危机波动模型的共同上限。

若未来进入 **Phase 0B**，同一逻辑框架可以被重新应用，但那将构成一轮新的评估：届时候选模型将替换为新的 crisis architecture，而 comparator 的定义也必须在新一轮协议中重新锁定。Phase 0A 与 Phase 0B 不是同一次评估的两个面，而是两次独立评估。

### 依据（阈值法律性质总说明）
以下阈值表中出现的数值（如 `0.05`、`2%`、`+0.10`、`+5%`）在法理上属于**治理阈值（Governance Thresholds）**，而非统计显著性阈值。它们不声称来自理论最优或数学推导，其合法性来自“在观察结果前预先写死并一体适用”，目的在于阻断事后优化、数据捕捞与幸存者偏差。

### A. 资格门槛（硬否决）
- `sigma_blowup == 0` 且 `pathology == 0`
- 三窗口 `corr_next > 0` 且 `rank_next > 0`
- **方向性稳定性补充铁律**：point estimate 大于零，仅是未被立即淘汰的最低资格门槛，绝不是方向性已被稳健建立的充分证据。关于 bootstrap sign-stability 的数值阈值，当前不在 Phase 0A 中写死；但在进入 Phase 0B 之前，必须先完成该阈值的显式预注册。若未完成该预注册，则任何下一阶段架构实验（包含但不限于 override / MoE / hard switch）绝对不得立项。

### B. Benchmark 比较基准
候选模型的 `std(z)` 必须同时满足：
- 绝对差值低于基准模型至少 `0.05`
- 相对改善率 `>= 2%`

### C. 2020 危机窗改善刚性阈值
- 2020 样本的候选模型 `std(z)` 必须按 B 项规则，相对**预先锁定且在 Benchmark Delivery 后冻结的主 benchmark**取得改善。
- 其法律形式固定为：`std(z)_{candidate,2020} <= std(z)_{benchmark,2020} - 0.05`，且相对改善率 `>= 2%`。
- `std(z)_{benchmark,2020}` 的数值在 benchmark 跑完后填入并立即冻结；`0.05` 与 `2%` 不得因结果而调整。

### D. OOS 危机窗（2008）保底阈值
- 在 2008 盲测集中，候选模型 `std(z)` 绝对不允许超过 `Benchmark_2008_std(z) + 0.10`
- 超出即判 `Invalid`

### E. 常态窗允许退化极限
- 2017/2018 平稳期 `std(z)` 恶化量 `<= +0.10`
- 且 `CRPS` 恶化量 `<= +5.0%`

### F. 最终结论判定规则
- **Success**：满足 A + C + D + E
- **Invalid**：违反 A，或 D，或 E
- **Inconclusive**：满足 A、D、E，但未突破 C

## 5. Phase 0A 的对称性预注册结论分支

### 裁决
实验结果必须且只能落入以下五种对称分支之一。候选模型（如 T5）的判定必须严格接受第 4 节协议逻辑（Success / Invalid / Inconclusive）；基准模型（Benchmark）的评估必须剥离相对比较条款，直接依据其自身的绝对物理指标界定。

### 依据
#### 分支一：当前失败是现架构（T5）特有缺陷
- **协议触发条件**：至少一个预先锁定的 benchmark 在 2020 与 2008 均保持最低绝对可用性（`corr_next > 0`、`rank_next > 0`、`sigma_blowup == 0`、`pathology == 0`，且 2008 不触发其自身崩溃性失效）；同时，候选模型（T5）的协议判定结果落入 `Invalid` 或 `Inconclusive`。
- **约束**：证实当前信息集尚未耗尽，问题出在 T5 内部算子。允许立项重构 T5 内部算子或引入非线性特征。

#### 分支二：当前失败是连续危机波动模型的共同上限
- **协议触发条件**：T5 的协议判定结果落入 `Invalid` 或 `Inconclusive`；且所有预先锁定的 benchmark 在 2020 和/或 2008 均无法维持最低绝对可用性（例如方向性非正、出现 `sigma_blowup > 0`、或在其自身危机窗中发生崩溃性失效）。
- **约束**：如果 benchmark 本身也出现绝对意义上的失败，则不能把当前失败解释为 T5 架构特有缺陷。因此，不支持把 crisis architecture 包装成“当前架构缺陷的直接修复方案”。本分支确认后，不得在当前证据链上直接立项任何危机架构。但这不等于永久禁止所有危机架构——若未来仍要立项，必须将其作为推翻连续性假设的全新独立假说重新走立项流程，并接受更严格的证据要求。

#### 分支三：不同危机样本行为异质，不支持统一危机层假说
- **协议触发条件**：候选模型在 2020 样本取得 `Success`，但在 2008 盲测中落入 `Invalid`；或者，benchmark 家族在 2020 与 2008 呈现断裂式异质表现，即至少一个 benchmark 在一场危机中保持最低绝对可用性，而在另一场危机中出现方向性倒置或严重 blowup。
- **约束**：证明 2020 仅为不可泛化的特定微观结构异象。禁止全局“危机重写”设计，终止以此为目的的统一危机层假说立项。

#### 分支四：T5 已达协议成功，但被更简单连续 benchmark 材料性支配
- **协议触发条件**：T5 自身取得 `Success`；且至少一个预先锁定的 benchmark 在安全与方向性不劣于 T5 的前提下，于危机窗对 T5 形成材料性支配。这里的“材料性支配”固定定义为：benchmark 在危机窗 `std(z)` 比 T5 低至少 `0.05` 且相对改善率 `>= 2%`。
- **约束**：这表明当前信息集尚未耗尽，但 T5 的复杂度或结构效率没有产生对应收益。允许重新审视 T5 的设计原则、特征集与结构效率，但不支持以此为由直接进入 crisis architecture。

#### 分支五：T5 在现有信息集下已足够，无需继续修补
- **协议触发条件**：T5 自身取得 `Success`；且不存在任何预先锁定的 benchmark 对 T5 形成材料性支配。
- **约束**：确认 T5 在当前信息集与连续模型类中已达到可接受上限。任何进一步修改不得再以“修补当前架构缺陷”为由直接立项；若仍要推进，必须提出独立新假说，并承担举证责任翻转。

## 6. Trigger inventory 的合法性规则

### 裁决
任何外部信号在未经 Lead-Lag Audit 之前，仅为无效候选数据，禁止具备接入任何门控或专家网络的资格。

### 依据
1. 合法 Trigger 的定义：在多个不重叠危机区间内，其显著变动必须严格先于实现波动率的阶跃（Lead），且在平稳期保持低假阳性。
2. **Lead window 绑定条款**：任何 trigger audit 之前，必须先预注册一个与未来架构无关的**保守执行延迟上限** `L_max`（交易日），并要求未来任何架构的有效执行延迟均不得超过该上限。在此基础上，合法 lead window 固定为 `{-L_max, ..., -1}`。`lag = 0` 或正滞后无效；早于 `-L_max` 的更长提前量可以做描述性报告，但不得单独赋予 trigger 准入资格。若未预注册 `L_max`，则整个 trigger audit 无效。
3. Audit 底线：必须输出合法 lead window 内的交叉相关或等价审计结果，以及常态下的 false positive rate。
4. 旧有 VIX ratio 不能直接复用：在未完成审计前，任何候选 trigger 都只是候选，不是依据。

## 7. OOS protocol 的硬约束

### 裁决
样本使用规则必须在立项前硬编码到隔离墙中，任何违规窥探行为直接判决整个研究线无效。

### 依据
1. 2008 样本必须 100% 完整 hold-out，不参与任何训练、参数拟合或超参数选择。
2. 2000–2002 样本作为异质验证集，用于否决仅对 2020 有效的特征或架构。
3. 2020 样本允许 in-sample 暴露，但绝不允许作为唯一危机代理。
4. 任何 override / MoE 所谓“成功”的阈值，必须在 Phase 0A 阶段预注册，禁止实验后倒推。
5. 若 OOS（2008）中出现 `sigma_blowup > 0` 或 `corr_next < 0`，该路线直接判无效。
6. **证据边界声明**：2008 的 blind holdout 只对未来候选 crisis architecture 与 benchmark 比较构成真正的 OOS 约束；它**不**会追溯性地把 `Stage A = T5` 变成对 2008 的真正 OOS 模型，因为 T5 的选择过程已经暴露于后来的危机样本信息（尤其 2020）。因此，2008 的验证结果可以用来否决新候选路线，但不得被夸大解释为“Stage A 的通用性已被 OOS 证明”。

## 8. 哪些内容仍然只是待验证假说

### 裁决
以下陈述在未通过本协议设定的 OOS 验证之前，均不得作为架构设计的既定事实。

### 依据
1. 存在一个同质且可泛化的统一危机态。
2. 流动性/信用利差信号能够超前预测波动率断层。
3. MoE 或 hard switch 能从根本上解决 overshoot。

## 9. 当前不能进入下一阶段架构实验的原因

### 裁决
在缺乏边界与对标的条件下直接进入 crisis architecture，不是在突破上限，而是在 2020 的局部样本里进行高自由度随机游走。

### 依据
- 未完成 benchmark 对照
- 未完成 cross-crisis validation
- 未完成 trigger lead-lag audit
- 未完成 bootstrap sign-stability 预注册

上述任一缺失，都足以使下一阶段立项失去方法论资格。

## 10. Phase 0B 的立项资格条件

### 裁决
未按顺序提交并通过以下三项交付物前，Phase 0B 禁止立项。

### 依据
1. **Benchmark Delivery**：在相同窗口上输出 EGARCH / GJR-GARCH 等锁定 benchmark 的完整诊断表。
2. **Trigger Audit Delivery**：提交通过 lead-lag audit 的候选 trigger 名录；若无一通过，则冻结所有 hard switch 构想。
3. **OOS Freeze Declaration**：在代码层面提交 2008 blind holdout 与 2000–2002 validation 的硬隔离方案，并预注册 bootstrap sign-stability 阈值与其他通过标准。
